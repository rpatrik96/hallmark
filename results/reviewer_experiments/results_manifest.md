# HALLMARK results manifest — post-relabel authoritative source of truth

Single source of truth for the paper-editing stage. Every number below is OLD (currently
printed in `hallmark-paper`) -> NEW (post-relabel, branch `fix/dev-public-mislabel-audit`).
Do NOT transcribe from anywhere else. Generated by the delta-eval pipeline in
`results/relabel_delta/` (no full re-runs; tool predictions are fixed per entry, only the
ground-truth label changed). Decimal convention matches the paper: cells drop the leading
`0.` in the LaTeX, but here values are shown with the leading `0.` for clarity.

## 0. Pipeline / provenance (how each cell was produced)

- **offline** = full-coverage per-entry prediction file re-scored against NEW labels via
  `hallmark.evaluation.metrics.evaluate()` (`results/relabel_delta/_rescore.py` +
  `regenerate_offline.py`). Exact. OLD recompute reproduces the published primary metrics
  within 3e-3 (asserted in the harness), so the NEW recompute is trustworthy.
- **delta-eval (reconstruct)** = summary-only tool (no per-entry file). New confusion matrix
  reconstructed from the published OLD aggregate + the tool's verdicts on the moved keys
  (`regenerate_summary.py` / `regenerate_cascade.py`, `_reconstruct.py`). DR/FPR/F1/MCC exact;
  TW-F1 within <=0.2pp; ECE held at published value (a <=27/1119 correctness flip moves ECE
  <1pp and adaptive binning can't be reproduced summary-only).
- All regenerated aggregates were written to `data/v1.0/baseline_results/*.json`.
- CIs: `results/relabel_delta/cis.json` (+ `cis_table.md`), stratified bootstrap, seed=42,
  n_bootstrap=10,000, computed on the NEW labels from per-entry predictions only.
- Split label counts (NEW): dev_public 513 VALID / 606 HALL (n=1119); test_public 312 / 519
  (n=831); test_hidden 210 / 244 (n=454); stress 1 / 121 (n=122); crossdomain 200 / 300 (n=500).

### Which entries actually move a tool score
Only HALLUCINATED->VALID moves change a tool's confusion matrix. The dataset-statistics
relabel moved **27 dev + 25 test** entries HALL->VALID (these drive `tab:stats`). The brief's
`relabel_flips.json` lists a 10 dev + 10 test + 9 hidden subset of score-changing flips it
chose to highlight; the regenerated tool aggregates use the **full** per-split moved-key sets
(`results/relabel_delta/changed_keys_*.json`: 27 dev / 25 test) because every HALL->VALID move
flips that entry's contribution to every tool. Retypes/valid_cleanup/unresolved do not change
scoring. **Verified self-consistent:** every offline tool's OLD recompute matches the printed
table to 3 decimals, and the NEW recompute matches what is tabulated below.
