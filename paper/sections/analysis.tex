\section{Analysis}
\label{sec:analysis}

\paragraph{Pre-screening effect.}
\label{sec:prescreening}
\textsc{Hallmark} includes an optional pre-screening layer---DOI format validation, year-range bounds, and author-name heuristics---available to all baselines.
Pre-screening provides incremental lift (${\sim}5$pp overall, ${\sim}18$pp on Tier~1) by catching obvious formatting issues before external API calls.
We report results both with and without pre-screening for transparency.

\paragraph{Calibration.}
\label{sec:calibration}
GPT-5.1 achieves the lowest ECE (0.107); open-weight LLMs show ECE 0.25--0.33, reflecting miscalibration from aggressive flagging.
Confidence calibration is critical for venue deployment, where reviewers need to trust flagged entries without manual verification.

\paragraph{Deployment PPV.}
\label{sec:ppv}
At venue-realistic prevalence ($\pi{\approx}2\%$), positive predictive value $\text{PPV} = \text{DR}\cdot\pi / (\text{DR}\cdot\pi + \text{FPR}\cdot(1{-}\pi))$ drops sharply: GPT-5.1 achieves only 8.7\% PPV (10 of 11 flagged entries are false alarms), and aggressive models fall below 3\% despite ${\geq}87\%$ DR (\cref{app:ppv}).
\emph{FPR reduction, not DR maximization, is the critical deployment metric}.

\paragraph{Cost--accuracy tradeoff.}
\label{sec:cost}
\cref{fig:cost} plots detection rate against cost.
DOI-only is cheapest (1 API call) but achieves only 25.6\% DR.
HaRC and verify-citations are bottlenecked by rate limiting ($<$7\% coverage).
GPT-5.1 achieves $3{\times}$ the detection rate at \$0.21/entry.

\paragraph{Fair-budget agentic comparison.}
\label{sec:agentic}
The zero-shot LLM baselines are compute-asymmetric with \texttt{bibtex-updater}: they make a single API call per entry, while \texttt{bibtex-updater} performs several lookups.
We therefore added agentic variants of GPT-5.1 that can issue up to five tool calls per entry---either to a multi-source suite (\texttt{resolve\_doi}, \texttt{search\_crossref}, \texttt{search\_openalex}, \texttt{search\_arxiv}; see \cref{app:llm-setup}) or to \texttt{bibtex-updater} exposed as a single tool.
Tool access lifts DR from 0.797 to 0.956--0.965, nearly matching \texttt{bibtex-updater}'s 0.946, but FPR rises from 0.171 to ${\approx}0.46$ because absence of evidence in any one source is interpreted as evidence of fabrication; F1 ends up below the zero-shot baseline despite the DR gain.
The BTU-as-tool variant slightly dominates the multi-source variant (DR 0.965 vs.\ 0.956, ECE 0.113 vs.\ 0.165), suggesting that a distilled upstream verdict is easier for the LLM to interpret than raw API responses, and that the marginal value of an agentic harness over simply piping \texttt{bibtex-updater} output into the prompt is small.

\paragraph{Infeasibility of HaRC even with a paid Semantic Scholar key.}
\label{sec:harc_feasibility}
To test whether HaRC's ${<}$7\% coverage was merely a rate-limit artefact, we re-ran HaRC on \texttt{dev\_public} with an authenticated Semantic Scholar API key (1 RPS guaranteed) and a 5-hour wall-clock budget.
Even so, \texttt{harcx}'s per-entry latency (${\sim}90$~s, dominated by internal DBLP and Google Scholar fallbacks that do not benefit from the S2 key) exceeds the 30-minute per-batch timeout: 0 of 1119 entries were checked by \texttt{harcx} itself, and the ``full-coverage'' run reflects pre-screening alone (DR 0.209, which we therefore treat as the pre-screening ceiling).
We conclude that HaRC is not operationally feasible at benchmark scale on shared infrastructure, independent of S2 quota.

\paragraph{Synthetic vs.\ real-world representativeness.}
\label{sec:synth_vs_real}
KS tests comparing feature distributions across the seven types with both real-world and synthetic instances show 4 of 7 are statistically indistinguishable ($p > 0.05$).
The three divergent types differ primarily in author count (real: 1.5--2.0 vs.\ synthetic: 3.3--4.7), reflecting that real LLM hallucinations fabricate shorter author lists.
These results support using perturbation entries as controlled diagnostics while motivating expansion of the real-world sample.

\paragraph{Temporal robustness.}
\label{sec:temporal_robustness}
A 480-entry temporal supplement (330 valid from 2024--2025 DBLP, 150 hallucinated) reveals that LLM verifiers degrade sharply on papers beyond their training cutoff (\cref{app:temporal_supplement}).
GPT-5.1's FPR rises from 17.1\% to 72.6\% ($4.2{\times}$); all five models show F1 collapsing to ${\approx}0.53$ (near-chance).
Detection rates simultaneously inflate to ${\geq}96.7\%$, confirming a universal ``flag everything unfamiliar'' failure mode.
Models with low baseline FPR suffer the largest relative degradation ($r = -0.82$, $p = 0.02$, $n = 7$), because conservative models rely more heavily on parametric memory confirmation.
Augmenting GPT-5.1 with API verification does not help: databases also lack recent papers.
A 60-entry probe first identified this pattern (\cref{app:temporal}); the supplement confirms it with $11{\times}$ more valid entries.

\paragraph{Failure modes.}
\label{sec:failure}
API tools fail on venue-level hallucinations (\texttt{preprint\_as\_published}, \texttt{wrong\_venue}) because bibliographic APIs do not distinguish venue publication from arXiv availability.
GPT-5.1 handles these (83--84\%) but struggles with \texttt{author\_mismatch} (63\%) and \texttt{near\_miss\_title} (63\%), which require exact bibliographic recall.
Combining bibtex-updater output with GPT-5.1 improves calibration (ECE: 0.107$\to$0.086) but not detection (DR: 79.7\%$\to$81.8\%)---the LLM overrides tool findings when they conflict with parametric judgment (\cref{app:codesign}).
The augmented system (GPT-5.1 + bibtex-updater) achieves the best ECE on the full benchmark (0.086), nearly halving GPT-5.1's standalone calibration error, while trading only ${\sim}3$pp of detection rate (81.8\% vs.\ 79.7\%).
This makes it the strongest option when calibration matters more than raw recall---for instance, in venue-scale deployment where reviewers act on confidence scores directly.

\paragraph{Water-filling effect.}
\label{sec:water_filling}
Benchmark optimization creates incentives to focus on easy subtasks: if Tier~1 hallucinations are easier to detect and contribute equally to the aggregate, tool developers have no incentive to invest in Tier~3.
We quantify this using the Gini coefficient of per-tier detection rates~\citep{hardt2025benchmarks}.
API tools show extreme concentration (Tier~1/3 ratio $>$30$\times$, Gini\,=\,0.65); LLMs show near-uniform performance (ratio 1.1--1.5$\times$, Gini\,$<$\,0.05).
We report Tier~3 F1 alongside the aggregate to prevent easy-type inflation.

\paragraph{Factor structure.}
\label{sec:factor_structure}
PCA on the tool$\,\times\,$type detection rate matrix (\cref{app:factor_analysis}) shows the first component captures the API-vs-LLM divide, while subsequent components capture recall--precision tradeoffs and tier-specific strengths---indicating that the tiered design adds orthogonal signal beyond a single factor.
